### What is Git?  
- Git is a **powerful system constantly tracking every change in files**, recording:  
  - What changed  
  - When it changed  
  - Who changed it  
  - Where the change happened  
- Supported file types: **Code, text, images, videos, almost any file format**.  
- Git creates **multiple versions of files, enabling easy rollback** to any prior state without losing data.  
- Git is widely used in coding projects but beneficial for any file versioning needs.

### Git Use Case and Importance  
- Scenario: After making changes your client dislikes, Git lets you revert to earlier file versions safely.  
- Git is a **version control system that safeguards every version**, making lost work a thing of the past.  
- Created by Linus Torvalds (Linux creator), Git is a long-lasting, **highly useful professional tool**.  
- Git consists of a **small set of simple commands that are straightforward to learn**, though initially it can seem intimidating.

### Difference Between Git and GitHub  
- **Git and GitHub are connected but distinct**: Git is the underlying tool (coffee), GitHub is the hosting service (coffee shop).
- **Git runs locally** on your computer, tracking project changes on your device.  
- GitHub is a **central online server (remote repository)** enabling team collaboration by hosting shared project copies.  
- Multiple developers working locally produce separate versions; GitHub **merges and synchronizes these into a single remote repo**.  
- Alternatives to GitHub include GitLab and Bitbucket, but **GitHub remains the most popular platform, owned by Microsoft**.  
- GitHub excels in **open source contributions and community-driven projects**.

### Git Architecture: Local vs Remote  
- Git has two major parts:  
  - **Local**: Your working directory on your machine where files exist and changes happen.  
  - **Remote**: Cloud storage (e.g., GitHub) where work is pushed or pulled for collaboration.  
- Core local workflow stages:  
  1. **Working directory:** The active project folder with all files.  
  2. **Staging area:** An intermediate zone where changes are prepared before final saving.  
  3. **Local repository:** Permanent storage of committed snapshots on your computer.
  
### Git Workflow in Detail  
- After code changes in the working directory, you **stage** changes via `git add` to prepare for saving.  
- Once confident, you **commit** staged changes (`git commit`) to the local repository, permanently recording the state.  
- Git creates a hidden folder `.git` to internally track all history and metadata.  
- Summary of local Git workflow:  
  $$ \text{Working Directory} \rightarrow \text{Staging Area} \rightarrow \text{Local Repository} $$

### Pushing to Remote Repository  
- When collaboration or backup is needed, one performs a **push** to upload commits from the local repo to the remote repo on GitHub.  
- Similar to cloud tools like Google Drive, this ensures code is safe online and accessible from anywhere.  
- Remote repo acts as **centralized hub for team collaboration and version consistency**.

### Installing Git  
- Git is downloadable from the official site with versions for Windows (32/64-bit), MacOS, and Linux.  
- Mac users can also install via Homebrew; Linux users follow distribution-specific instructions.  
- After installation, verify with `git --version` in terminal to check Git’s presence and version.  
- Terminal environments:  
  - Windows: Command Prompt, PowerShell, or Git Bash (preferred for its Linux-like feel).  
  - MacOS/Linux: Terminal app.

### Setting Up a Project and Local Repository  
- Navigate within the terminal using commands like `cd`, `mkdir`, `touch`.  
- Create a project folder (`mkdir foldername`), enter it (`cd foldername`), and create files/folders.  
- Writing simple content to files prepares them for Git tracking.  
- Initialize Git repo with `git init` to start tracking the folder—creates hidden `.git` directory.  
- `.git` directory stores all Git’s internal data (commits, changes, authors).


### Create a New Remote Repository  
-  Log in to GitHub (or GitLab/Bitbucket).
- Click on the **New Repository** button.
- Give your repository a name (e.g., `project`).
- Choose public or private, then click **Create repository**.
- You now have a remote repository URL (e.g., `https://github.com/username/project.git`). 
- Clone the remote repo locally using `git clone <repo-URL>`.  
- This creates a copy of remote repo on your machine with full Git tracking.  

### Connect Local Repository to Remote
- In your local repo folder, run: `git remote add origin `.
- Example: `git remote add origin https://github.com/username/project.git`.
- Verify connection with: `git remote -v`.
- Both local and remote repositories coexist and can synchronize changes.

### Tracking File Changes - `git status`  
- Modifying files is detected automatically by Git.  
- Running `git status` shows modified, new, or deleted files that are untracked or staged.  
- Helps developers **monitor the project's current status** at any point.


### The Staging Area - `git add` Command  
- Moving changes from working directory to staging area is called **adding**.  
- Variations to add files include:  
  

| Command        | Effect                                  | Scope                       |
|--------------------|-----------------------------------------|-----------------------------|
| `git add -A` or `git add --all`   | Stage all changes (new, modified, deleted) | Entire project              |
| `git add .`    | Stage changes within current directory   | Current directory + subfolders  |
| `git add *.txt`| Stage all files by extension pattern       | Current directory only       |
| `git add *`    | Stage new & modified files, excludes deleted files | Current directory only       |


### Committing Changes - `git commit`  
- Commit permanently saves staged changes to local repo with a descriptive message using `git commit -m "message"`.  

- Before making the first commit, Git requires the **user identity to be configured**:
  - `git config --global user.email "you@example.com"`  
  - `git config --global user.name "Your Name"`  
- Committed changes become part of project history and can be reviewed or reverted later.
 
### Undo Last Commit  
- With `git reset HEAD~` (or related), you can undo last commit, moving changes back to working directory unstaged.  
- Gives ability to revise or redo commits as needed.
  
### Removing Files - `git rm`  
- The `git rm` command simplifies file deletion by removing the file and staging the deletion in a single step (e.g., `git rm <file>`).  
- To force delete files with uncommitted changes: use `git rm -f <file>`.  
- To remove files from staging without deleting: use `git rm --cached <file>`.  
- Recursive removal of folders with contents: `git rm -r <folder>`.


### Viewing Commit History - `git log`  
- `git log` shows full commit history with details and unique commit IDs.  
- `git log --oneline` presents a concise summary with commit IDs and messages.  
- Commit IDs can be used to refer to specific commits for revert or checkout.

### Branching and Merging  
- Git branching allows **parallel lines of development** that are isolated from the main branch. 
- The default branch is typically called `main` or `master`. 
- To rename the current branch to `main`, use: `git branch -M main`
 
- Workflow:  
  1. Create branch (`git branch <branch-name>`)  
  2. Switch branch (`git checkout <branch-name>`)  
  3. Make changes, stage them, and commit them on the feature branch.
  4. After testing, switch back to the main branch and merge the feature branch into it.

- Switching branches updates your working directory to reflect the files and changes associated with the selected branch.
- Changes committed on one branch remain exclusive to that branch and will not appear in other branches until they are merged.

- A useful analogy is a restaurant's main kitchen and test kitchen: developers can experiment safely in a separate branch without affecting the main codebase. 


### Branch Merging Commands  
- To merge changes from Branch B into Branch A:
  1. Switch to Branch A: `git checkout main`
  2. Merge Branch B into Branch A: `git merge <branch-name>`.
- Merging combines the changes from two branches while preserving the commit history of both branches.
- Merge conflicts occur when the same parts of a file have been modified differently in each branch. These conflicts must be resolved manually before the merge can be completed.


### Merge Conflicts and Resolution  
- Merge conflicts occur when changes made in different branches overlap or modify the same section of a file.
- Git marks the conflicting areas within the file, displaying both versions of the changes.
- The developer must review the conflicts and decide which changes to keep, modify, or combine manually.
- After resolving all conflicts, stage the affected files and create a commit to finalize the merge.
- Merge conflicts are common in collaborative development environments and should be resolved carefully to avoid introducing errors or losing important changes.


### Checking Out Previous Commits  
- By using a commit ID (`git checkout <commit-id>`), you can view the state of the project at any previous commit.
- Checking out a specific commit places Git in a detached HEAD state, allowing you to explore historical versions without switching branches.
- While in a detached HEAD state, any new commits are not associated with an existing branch unless you create a new branch from that point.
- To return to the latest version of the main branch, use: `git checkout main`.
- This feature provides **full control over project history**, making it easy to inspect, test, or debug previous versions of the project.

  
### Comparing Commits - `git diff`  
- The `git diff <new-commit-id> <old-commit-id>` command displays the differences between two commits, showing which lines were added, modified, or removed.
- The output is color-coded for easier interpretation:
  - Red indicates lines that were removed.
  - Green indicates lines that were added.
- The order of the commit IDs affects how the changes are presented. Typically, the newer commit is listed first to view the changes relative to the older commit.
- `git diff` is a useful tool for reviewing code changes, tracking modifications, and understanding the evolution of a project over time.
 
### Synchronizing with Remote: Push, Fetch, Pull  
| Command       | Action                                  | Effect                                 |
|---------------|----------------------------------------|---------------------------------------|
| `git push`    | Send local commits to remote repo      | Uploads commits to remote branch       |
| `git fetch`   | Downloads updates from the remote repository without merging them| Updates local repo references only     |
| `git pull`    | Fetch + merge remote updates           | Updates the working directory with remote changes |

- Example: `git push origin main`. This command pushes local `main` branch to remote repository.  
- Each branch must be pushed individually (e.g., `git push origin <branch-name>`).  
- The `git fetch `command allows you to inspect remote changes before integrating them into your local branch.
- The `git pull` command combines `git fetch` and `git merge` into a single operation, making it a convenient way to synchronize your local repository with the latest remote updates.
- These commands are essential for collaborating with other developers and keeping local and remote repositories in sync.


### Undoing Local Changes - `git restore`  
- Restore files or directories back to last committed state, discarding uncommitted changes.  
- Usage:  
  - `git restore <file>` for single file  
  - `git restore .` for entire repo  
  - Use `git restore --staged <file>` to unstage changes but keep files modifications intact.
- Use this command carefully, as restoring a file will permanently discard any uncommitted changes in that file.

### Temporarily Saving Work - `git stash`  
- This command temporarily saves uncommitted changes, allowing you to switch branches or perform other tasks without creating a commit. 
- Commands:  
  - `git stash`: Save changes to stash and revert working directory.  
  - `git stash pop`: Restore latest stash and remove it from stash list.  
  - `git stash apply`: Restore stash but keep it listed for reuse.  

- Stashes are identified by names such as `stash@{0}`, `stash@{1}`, and so on. 
- `git stash list`: View all saved stashes.
- `git stash drop stash@{0}`: Stash drop removes stash entries.


### Reverting Commits - `git revert`  
- The `git revert` command creates a new commit that **reverses the changes introduced by a specified earlier commit** without removing it from the project history.
- This approach preserves the complete commit history, making changes easy to trace and audit (`git revert <commit-id>`).
- Unlike `git reset`, which removes commits and rewrites history, `git revert` safely undoes changes while maintaining a consistent history.
- Because it does not alter existing commits, `git revert` is the recommended method for undoing changes in public or shared repositories.
- Using `git revert` helps maintain a clear, consistent, and collaborative project history.

### Rebasing Branches - `git rebase`  
- Rebasing **moves the base of the current branch to the updated main branch**, replaying commits on top for a linear history.  
- This creates a **cleaner, more linear commit history** by replaying commits sequentially rather than creating a merge commit.
- Alternative to `git merge` which creates extra merge commits.  
- Cleaner commit history made by replaying commits sequentially.  
- Not recommended for shared branches unless all collaborators coordinate, because it rewrites commit history.  
- Usage: While on feature branch, `git rebase main`.

**Merge vs. Rebase**

|  Feature    | 	`git merge`	   |    `git rebase` |      
|---------------|----------------------------------------|---------------------------------------|
History	| Preserves branch structure	|Creates a linear history |
Merge Commit |	Creates a merge commit	| No merge commit by default |
Commit History	 | More detailed	| Cleaner and easier to read |
Shared Branches |	Safe for collaboration	 | Can cause issues if history is already shared |

- Use **merge** when preserving branch history is important.
- Use **rebase** when you want a clean, linear project history.


### Pull Requests (PRs) on GitHub  
- Pull requests are **requests to merge changes from one branch into another** (usually feature branch into main).  
- Facilitates **code review and collaboration**, granting controlled merges.  
- Workflow:  
  1. Push branches to GitHub  
  2. Open pull requests via GitHub UI  
  3. Review comparisons (files changed, commits)  
  4. Discuss and approve  
  5. Merge the PR to update main branch  
- Pull requests keep main branch stable and provide a collaborative review system.
